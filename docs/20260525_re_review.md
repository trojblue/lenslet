I re-reviewed `trojblue/lenslet` at `fa6ebbbee815a70289a50d887a6c4f3fed68255a`. I inspected the current GitHub source and commit history; I could not reliably clone/run it in this environment, so this is a source-level, interaction-focused review rather than a live visual QA pass. Treat the ‚Äúlikely glitch‚Äù items as things to verify with browser tests.

## Overall verdict

The repo is **less sloppy than the previous baseline**, but the remaining problems are the kind that make software still *feel* half-polished: race conditions, responsive affordances that are declared but not guaranteed, duplicated interaction engines, uncoalesced pointer/resize updates, and global keyboard behavior that works until the user does something slightly outside the happy path.

Since the earlier `3210957` baseline, the commit history shows targeted work on viewer hit targets, pan/click interactions, viewer open-path stabilization, a viewer flicker browser baseline, hover preview restoration, contained overlays, viewport menus, justified row layout, and comparison-export cleanup. That is real progress, but it also suggests many fixes are still patch-like and interaction-specific rather than fully absorbed into a unified frontend reliability model. ([GitHub][1])

The README now explicitly promises touch/mobile behavior: tap-first browse, visible grid/folder action buttons on coarse pointers, narrow-screen select mode, and in-viewer next/previous controls on phone widths. Those promises are exactly where I would focus the next pass, because several source-level details still look easy to regress or possibly incomplete. ([GitHub][2])

## Highest-priority lingering issues

### 1. The mobile viewer navigation looks like it may be dead or under-wired

`Viewer.tsx` renders a `.viewer-mobile-nav` block with Prev/Next buttons when `onNavigate` exists. ([GitHub][3]) But the source CSS sets `.viewer-mobile-nav { display: none; }`, and the same CSS block shown around it does not include an enabling media rule. ([GitHub][4]) That directly conflicts with the README promise that viewer navigation remains available at phone widths. ([GitHub][2])

**Fix direction:** make this explicit and testable. Add a source-level rule such as ‚Äúvisible below phone breakpoint‚Äù or ‚Äúvisible on coarse pointers,‚Äù not a hidden dependency on generated CSS or future Tailwind classes. Then add a Playwright regression at something like `390x844` verifying the buttons are visible and actually navigate. This is a polish issue, but it is also reliability: mobile users cannot use what they cannot discover.

---

### 2. Programmatic zoom requests can be swallowed before the image/container is ready

`Viewer.tsx` consumes `requestedZoomPercent` regardless of whether the zoom operation succeeds: it calls `zoomToPercent(requestedZoomPercent)` and then immediately calls `onZoomRequestConsumed`. ([GitHub][3]) But `zoomToPercent` returns `false` when the container or image size is unavailable. ([GitHub][5]) That means a toolbar zoom request can be lost if the user clicks quickly during load, after navigation, or during a weird resize/open path.

**Fix direction:** only consume the request when `zoomToPercent` returns `true`; otherwise keep it pending until `activeResource`, `imageReady`, and measured geometry are available. Add a regression: open viewer on a slow image, immediately request 200%, wait for load, assert the final transform/zoom label actually reflects 200%.

This is exactly the sort of thing that makes an app feel ‚Äúcheap‚Äù: the control accepts an action, but nothing happens, and the state quietly forgets the intent.

---

### 3. Viewer pan/zoom still uses React state as a high-frequency motion pipeline

`useZoomPan` stores transform refs, but each transform application still pushes four React state updates: base, scale, tx, and ty. ([GitHub][5]) Pointer move and pinch handlers call `applyTransform` directly during high-frequency pointer streams. ([GitHub][5]) Compare has the same pattern, with even more state updates for two images. ([GitHub][6])

This can work on a good machine, but it is not the most robust model for ‚Äúsmooth under weird proportions, half-screen resizing, high-refresh trackpads, and touch.‚Äù `requestAnimationFrame` is the right primitive for frame-coalesced visual updates because it schedules work before the next repaint and generally tracks display refresh rate. ([MDN Web Docs][7])

**Fix direction:** move transform rendering to a single frame-scheduled pipeline. Keep canonical transform in refs, schedule one RAF, and either set a single `{base, scale, tx, ty}` state object once per frame or imperatively update a CSS transform on the image element while mirroring state only when needed for UI labels. Also cancel pending RAFs on unmount; `cancelAnimationFrame` exists for exactly that purpose. ([MDN Web Docs][8])

---

### 4. Resize preservation is close, but it can still snap because pan-slack semantics are inconsistent

The viewer uses `VIEWER_PAN_SLACK` for wheel zoom/pinch/pan clamping. ([GitHub][5]) The shared transform helper supports slack via `ImageTransformClampOptions`, and `clampImageOffset` behaves differently when `panSlack` is enabled. ([GitHub][9]) But `restoreImageTransformForCenter` does not accept or pass clamp options; it always ends with a strict `clampImageTransform(...)`. ([GitHub][9]) The resize observer path calls `restoreImageTransformForCenter`, so a user who has panned within slack can get pulled/snapped during resize. ([GitHub][5])

**Fix direction:** either pass clamp options through `restoreImageTransformForCenter`, or remove slack at fit-scale entirely so the behavior is conceptually simpler. Right now the viewer says ‚Äúyou may pan a bit‚Äù during normal interaction, then resize can say ‚Äúactually, no.‚Äù That discrepancy is a prime source of ‚Äújittery/glitchy‚Äù feel.

---

### 5. ResizeObserver work is uncoalesced and not canceled

The viewer and compare zoom hooks create a `ResizeObserver` and call `requestAnimationFrame(() => preserveCenterAfterResize())`, but they do not store the RAF id, coalesce multiple resize notifications, or cancel the pending callback in cleanup. ([GitHub][5]) MDN notes that `ResizeObserver` reports element dimension changes, that resize callbacks are part of a sensitive pre-paint/layout cycle, and that deferring work with RAF is a known pattern when avoiding resize-loop issues. ([MDN Web Docs][10])

**Fix direction:** store `resizeRafRef`; if one is pending, do not schedule another; cancel it on cleanup. This is small, but it reduces ‚Äúhalf-screen drag the browser edge and the image swims around‚Äù failure modes.

---

### 6. Compare viewer can reset the user‚Äôs view after one image loads late

In `CompareViewer`, both `markImageAReady` and `markImageBReady` call `fitAndCenter()`. ([GitHub][11]) Since A and B can load at different times, the second image to finish can refit/recenter the compare stage after the user has already started inspecting the first loaded side. The compare hook also fits/restores both images together. ([GitHub][6])

**Fix direction:** use a pair-generation model. When `{aPath,bPath}` changes, reset ‚Äúinitial fit pending.‚Äù Fit only once after both current full-size images have valid dimensions, or fit once on first usable dimensions but freeze that initial fit after any user gesture. Track `hasUserInteractedSincePairChange`; if true, never auto-fit from an image-load callback.

This is one of the most important ‚Äúpolish‚Äù fixes because compare mode is where users are likely zoomed in and detail-sensitive.

---

### 7. Global keyboard handlers are too broad and inconsistent

The image viewer installs a `window` keydown listener for Arrow/A/D navigation and does not check whether focus is inside a text input, button, menu, or contenteditable area. ([GitHub][3]) Compare does ignore inputs, but still listens globally and still treats `a`/`d` as navigation when focus could be on a button or another control. ([GitHub][11]) WAI‚Äôs modal dialog pattern expects focus to stay within the dialog, Escape to close, and dialog behavior to be coherent for all users. ([W3C][12])

**Fix direction:** make one shared `useDialogNavigationKeys` hook. Scope it to the active dialog root, ignore text inputs/buttons/menus/contenteditable and modifier-key combos, and make it active only while that dialog is topmost. Keep Escape in the focus-trap hook. This removes a lot of accidental ‚ÄúI pressed D in a control and the image changed‚Äù weirdness.

---

### 8. Touch action buttons are still visually hover-first in source CSS

The README promises explicit grid/folder action buttons visible on coarse pointers. ([GitHub][2]) The grid action button exists in `VirtualGrid.tsx`, with `className="grid-item-action-btn touch-manipulation"`. ([GitHub][13]) But source CSS initializes `.grid-item-action-btn` at `opacity: 0` and only reveals it on group hover or focus-visible. ([GitHub][4]) The folder tree action buttons show a similar hover/focus pattern. ([GitHub][4])

**Fix direction:** add an explicit coarse-pointer rule, for example showing these buttons under `@media (pointer: coarse)` or while a touch-selection mode is active. Then add a browser test that emulates touch/coarse pointer and asserts action buttons are visible without hover. ‚ÄúWorks on desktop hover‚Äù is not enough for a touch promise.

---

### 9. `VirtualGrid.tsx` remains a god component

`VirtualGrid.tsx` is 865 lines / 800 LOC / 32.2 KB. ([GitHub][13]) It currently combines virtualization, adaptive row rendering, selection, keyboard navigation, long-press, scroll state, preview hover, thumbnail loading coordination, drag ghost behavior, and action menus. The component is not random, but it is too much surface area for a single file if the goal is extendable and reliable.

There are already signs of this stacking: custom scroll animation lives in the component, scroll idle state lives there, preview timers live there, long-press controller wiring lives there, and the preview hotspot has inline decorative CSS including a path clip, gradient, shadow, and blur. ([GitHub][13])

**Fix direction:** split behavior, not just JSX. I would extract:

`useGridSelection`, `useGridKeyboardNavigation`, `useGridScrollState`, `useHoverPreview`, `useLongPressActions`, `useDragGhost`, and a dumb `GridCell`/`GridRows` rendering layer.

The aim is not ‚Äúclean code aesthetics‚Äù; it is making each interaction testable without a full virtual grid fixture.

---

### 10. Scroll and animation cleanup has small but real leaks/glitches

`VirtualGrid`‚Äôs scroll listener sets a timeout to mark scrolling idle, but cleanup removes only the scroll listener and does not clear the timeout. ([GitHub][13]) Its custom scroll animation cancels the previous RAF when starting a new one, but I did not see an unmount cleanup for `scrollAnimRef`. ([GitHub][13])

**Fix direction:** clear the idle timeout and cancel any pending RAF in cleanup. This is not glamorous, but these are the edge cases that show up as ‚ÄúReact state update after unmount,‚Äù stale preview behavior, or scroll fighting when switching modes quickly.

---

### 11. Compare divider drag is manually wired and brittle under resize/lost capture

`handleDividerPointerDown` reads the stage rect once, calls `target.setPointerCapture`, then installs `window` pointer listeners that compute split percentage from that stale rect. ([GitHub][11]) It removes listeners on pointerup/cancel, but there is no component-level cleanup if the compare viewer unmounts mid-drag, and the rect will be wrong if the stage resizes during drag. Pointer capture is appropriate here‚ÄîMDN describes it as targeting future pointer events to the captured element until release or pointerup‚Äîbut the implementation should treat capture loss/unmount/resize as normal paths, not rare exceptions. ([MDN Web Docs][14])

**Fix direction:** extract `useDividerDrag`, keep listener cleanup in an effect, handle `lostpointercapture`, recompute rect on move or subscribe to stage size, and RAF-throttle `setSplitPct`.

---

### 12. Motion/accessibility polish is still not encoded as a product rule

The CSS includes repeated animations such as `syncPulse`, `syncBreathe`, and `slideDown`. ([GitHub][4]) MDN‚Äôs `prefers-reduced-motion` guidance specifically calls out motion-based animations, including scaling/panning large objects, as potential vestibular triggers and documents the `reduce` preference. ([MDN Web Docs][15]) The viewer image also uses generic `alt="viewer"`, compare images use `alt="compare A"` / `alt="compare B"`, and focus outlines are suppressed on viewer/compare containers. ([GitHub][3])

**Fix direction:** add a global reduced-motion layer that disables nonessential infinite animations, long transitions, hover preview fades, and spinner alternatives where possible. Use file names or labels in image alt text where it helps; use empty alt only where images are purely decorative thumbnails behind a labeled UI. Restore meaningful focus visibility for dialog controls instead of blanket `outline: none`.

## Code debt that is worth paying down now

The good news: the repo is already using reasonable frontend primitives, including React, Vite, TanStack React Query, TanStack React Virtual, and dnd-kit. ([GitHub][16]) This does **not** need a fashionable stack rewrite. The debt is interaction architecture.

The most valuable refactor is to turn `imageTransform.ts` into the single transform engine for both viewer and compare. It is already a pure helper module with scale bounds, fit, clamp, zoom-around-point, pan, center capture, and restore-center logic. ([GitHub][9]) But viewer and compare still have separate hooks with similar pointer/resize/wheel machinery. ([GitHub][5])

I would create a shared `useImageSurfaceTransform` core with adapters:

```ts
useSingleImageSurfaceTransform(...)
useCompareImageSurfaceTransform(...)
```

The core should own pointer capture, RAF scheduling, resize coalescing, wheel normalization, center preservation, cleanup, and user-interaction flags. The adapters should only decide how many image layers exist and how transform output maps to DOM nodes.

## Test gaps I would close next

There are backend/server tests and at least one Playwright large-tree smoke test in the root test directory. ([GitHub][17]) The frontend package exposes `vitest run`, and the visible frontend test folders include hook/component-level tests such as `useZoomPan.test.ts` and `thumbUrlHook.test.ts`. ([GitHub][16]) The missing confidence is not ‚Äúthere are no tests‚Äù; it is that the most expensive regressions are black-box interaction regressions.

Add browser tests for:

1. **Phone viewport:** `390x844`, viewer Prev/Next visible and usable.
2. **Coarse pointer:** grid/folder action buttons visible without hover.
3. **Immediate zoom after open:** request 200% before image load; final zoom is honored.
4. **Navigate while image load is racing:** old image never flashes for the new path.
5. **Resize while zoomed and panned:** center is preserved without snap.
6. **Compare late-load:** user pans/zooms after A loads; B loading later does not reset view.
7. **Compare divider drag:** drag while viewport changes; split remains sane.
8. **Keyboard focus:** A/D do not navigate while focus is in controls/text fields.
9. **Reduced motion:** with reduced-motion emulation, nonessential animations are disabled.
10. **Pathological images:** very tall, very wide, tiny, broken image, slow image, transparent image.

---

## Suggested work order

**P0 ‚Äî fix visible/behavioral lies:** mobile viewer nav visibility, coarse-pointer action visibility, swallowed zoom requests, compare late-load reset.

**P1 ‚Äî remove jitter sources:** RAF-scheduled transform updates, coalesced/canceled ResizeObserver handling, consistent pan-slack during resize, divider drag cleanup.

**P2 ‚Äî make it extendable:** split `VirtualGrid`, unify viewer/compare transform hooks, centralize dialog keyboard handling, add reduced-motion and accessible labeling rules.

After that, the app should feel much less like ‚Äúworks at one aspect ratio/browser level‚Äù and more like a reliable gallery tool. The main theme is: stop treating interactions as one-off event handlers, and promote them into small tested state machines with explicit lifecycle cleanup.
