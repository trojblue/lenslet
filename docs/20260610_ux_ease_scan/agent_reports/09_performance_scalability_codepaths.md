# Performance/Scalability Codepath Scan

## Scope And Files Inspected

This scan focused on speed, stability, payload shape, request pressure, and render predictability in the browse/search/media hot paths. I inspected backend route layers, storage implementations, browse query evaluation, row-store access, caches, frontend React Query usage, blob/request budgets, virtualized rendering, bundled asset shape, and smoke/perf coverage.

## Concise Hot-Path Map

- Browse query path: `useAppDataScope` builds a `folder-query` infinite query, `api.queryFolder` posts to `/folders/query`, `build_folder_query` uses storage-native `query_browse_scope` when present, otherwise falls back to item materialization and `evaluate_browse_records`.
- Table browse path: `TableStorage.query_browse_scope` gets row ids from `TableRowStore`, builds `BrowseQueryRecord` objects for every row in scope, evaluates filters/search/sort, then materializes only the requested window.
- Recursive folder payload path: `/folders` and `/folders/query` both enter `src/lenslet/web/browse.py`; recursive no-limit payloads can still build and cache full snapshot windows up to the hard limit.
- Facet path: `/folders/facets` calls `facet_summary_for_scope` for table storage when available; fallback scans item snapshots.
- Search path: frontend toolbar search primarily uses browse query. The legacy `/search` route still exists and calls `storage.search` with loose `limit` validation.
- Media path: cards request `/thumb`, viewer/compare request `/file`; frontend blob LRU caches, request budget queues, backend disk thumb cache, and a LIFO thumbnail scheduler reduce duplicate work.
- Render path: `VirtualGrid` uses React Virtual for row virtualization; visible rows render through `VirtualGridRows`, with each `ThumbCard` lazily requesting thumbnails after intersection and scroll-idle checks.
- Large-tree validation path: `scripts/browser/large_tree/smoke.py` measures browse endpoints, request budget peaks, thumbnails, and responsiveness against baselines.

## 3 Quick Wins

1. Add bounded validation or a deprecation path for `/search` so loose limits and inconsistent legacy behavior stop being a performance footgun.
2. Start with tests for patching `folder-query` infinite pages, then replace broad browse-query resets for updates that can be patched safely.
3. Precompute table display-field extraction helpers so inspector selection no longer rebuilds duplicate/metric metadata per row.

## 3 Medium Projects

1. Build measured row-native browse query fast paths for table storage, starting with name/path/date sorts, cached text matching, and metric top-k windows.
2. Complete the hard cutover from recursive no-limit folder payloads to bounded query/count/facet endpoints, then enforce the API policy.
3. Add predictable cache and queue budgets: backend thumbnail byte-budget LRU, endpoint-specific request queue caps, and smoke-test thresholds for visible-work latency.

## Things Not To Do

- Do not raise hard limits or page sizes to mask 413s or slow recursive responses; reshape payloads instead.
- Do not add another frontend state manager for server data; React Query, blob caches, and targeted patch helpers are the right primitives.
- Do not memoize every grid component without profiler evidence; memoization can add stale-state bugs and code bulk.
- Do not build a broad SQL-like query engine before optimizing the small set of row-store browse operations users actually hit.
- Do not remove request budgeting or the LIFO thumbnail scheduler; tune queue semantics with evidence.
- Do not rely on `docs/agents_archive` as a current performance source of truth.

## Top 5 Recommendations

1. Patch `folder-query` infinite-query caches in place for live sidecar and metric updates, and stop resetting broad browse queries unless ordering or membership truly changes.
2. Measure and then optimize table `query_browse_scope` so common filters/search/sorts avoid full-scope record construction and full sorting.
3. Split metric population/ranking data from bounded item pages so metric sorting does not ship 50,000 item payloads to the first screen.
4. Remove remaining recursive no-limit UI/API paths and make paginated browse query, count, and facets the only large-tree path.
5. Add explicit memory and queue budgets for backend thumbnails and frontend request queues, validated by large-tree smoke metrics rather than assumptions.
